Від реактивної підтримки до Центру Знань
```
Схема «Було/Стало»
Місце для візуальної схеми потоку звернення: Користувач → L1 → L2 → Ескалація vs Користувач → База Знань → L1 (за потреби).
Було
Реактивний потік
Чати & тікети → ескалації, мало стандартизованих рішень.
Стає
Проактивний потік
База Знань вирішує типові кейси, L1 фокусується на винятках & покращеннях.
Поточна модель: «гасіння пожеж»
Як виглядає сьогодні
  • Велика частина запитів розв’язується «з нуля» словами в чаті або в тікеті.
  • Ескалації летять до більш досвідчених спеціалістів навіть для повторюваних кейсів.
  • Онбординг новачків базується на «усній традиції» та парному навчанні.
  • Знання розпорошені між особистими нотатками, старими листами та архівними документами.
Цільова модель: Центр Знань
Як має бути у 2026
  • Більшість стандартних кейсів закривається через актуальну Базу Знань.
  • Частка ескалацій ≤20%: ескалація тільки для справжніх «edge cases».
  • Онбординг новачків через готові сценарії, чеклісти та навчальні статті (−30% часу).
  • Чіткі ролі: хто створює, рев’ює та оновлює знання по кожному продукту/процесу.
Фокус для команди
Менше хаосу в чатах, більше стандартизованих рішень у Базі Знань.
Роль кожного
Кожен фахівець підтримки = автор або рев’юер статей по своїх зонах.
Очікуваний ефект
Швидші відповіді, менше перевантаження L2, прозорий онбординг.
```
